# Research: CRPG 核心机制设计 — 主题如何映射为玩法

> Parent Issue: #5
> Agent: research-agent
> Date: 2026-07-21

---

## 1. Problem Definition

### Current Behavior

项目当前处于脚手架阶段 — `gdscripts/main.gd` 仅显示 "Hello World"，`gdscripts/game_manager.gd` 是空的全局管理器，`scenes/main.tscn` 只有一个 Label。**尚无任何游戏机制、状态系统、对话系统或叙事映射层。**

### Expected Behavior

设计一套完整的 CRPG 核心机制系统，将「3 个月做卖座游戏的绝望感和行业困境」这一主题具象化为可玩的游戏机制。这套机制应满足：

1. **希望/绝望不是 HP/MP**，而是决定世界如何呈现的滤镜
2. **每个 NPC 对话不仅是叙事，是一次状态检定**
3. **雨夜作为环境本身就是一种压力系统**
4. **3 个月 deadline 应有时钟机制**

### User Scenarios

- **Scenario A（核心体验）:** 玩家扮演一个处于绝望边缘的游戏开发者，在雨夜城市中漫步。雨、NPC、回忆、地标 — 一切交互都在改变玩家的内心状态，而状态又改变世界呈现的方式。
- **Scenario B（关键互动）:** 玩家遇到 NPC，对话既是叙事推进也是一次"状态检定"— 选择如何回应不仅改变 NPC 对你的态度，更改变你对自己的认知（三组滑条的状态值）。
- **Scenario C（环境压力）:** 雨越下越大，街道越来越暗，deadline 越来越近 — 环境本身对玩家产生持续的压力，催促玩家"做点什么"。
- **Frequency:** 持续 — 每个交互、每个场景、每句对话都在运行这套机制。

## 2. Design Intent

### Why Does Current Behavior Exist?

项目刚刚初始化，尚未进入任何设计或实现阶段。当前只有 Issue #1（Hello World）已完成整个 pipeline。这是第一个设计层面的 Issue，也是整个游戏的基石。

### Why Change Now?

机制设计是所有后续决策的前置依赖。根据 `docs/RAW/game-to-issues-urban-night-walker.json` 的依赖链：

```
Issue #5 (CRPG核心机制) ── 基石 ──┐
                                   ├── Issue #2 (叙事架构)
                                   ├── Issue #4 (状态-世界反馈)
                                   ├── Issue #3 (海明威写作约束)
                                   └── 所有后续 feature 依赖
```

不先解决"希望是什么"、"绝望如何影响世界"、"选择为什么重要"这些问题，后续的叙事结构、对话引擎、场景设计都无法起步。

### Previous Constraints

- 引擎：Godot 4.7.1（非物理引擎 CRPG，侧重 UI、文本渲染、状态管理）
- 语言：GDScript 2.0（静态类型）
- 风格参考：极乐迪斯科（Disco Elysium）的系统即叙事方法论
- 视觉风格：Edward Hopper 文字渲染风格
- 写作风格：海明威式（短句、冰山理论）
- 时间限制：3 个月开发周期（这是**游戏内**也是**开发**的时间限制）
- 平台：macOS / Linux

## 4. Solution Comparison

### Approach A: 极乐迪斯科式 — 三轴人格系统即世界观滤镜

**Description:**

核心思路：将「希望-绝望」「热情-倦怠」「信念-动摇」三组滑条（每组 1-10，和为 10）作为世界认知的滤镜，而非传统的 HP/MP。

- **希望/绝望（Hope/Despair）：** 决定世界文本的冷暖色调。高希望（≥7）：便利店灯光更暖，NPC 语气更温和，雨声像"背景白噪音"。高绝望（≥4）：影子更长，NPC 更冷漠，雨声像"敲打骨头的鼓点"。
- **热情/倦怠（Vigor/Burnout）：** 决定行动力和交互可能性。高热情：探索选项更多，能主动发问，NPC 对话中会出现新话题。高倦怠：交互选项减少，被动等待，重要对话可能被跳过。
- **信念/动摇（Conviction/Falter）：** 决定对话检定的结果权重和内心独白的自信度。高信念：检定成功率 +1，内心独白坚定（"我知道该怎么做"）。高动摇：检定成功率 -1，内心独白自我怀疑（"也许我根本不该做游戏"）。

**机制设计：**
1. **对话即检定：** 每个关键 NPC 对话不是树状分支，而是状态值区间匹配。玩家当前的「热情」「信念」值决定对话进入哪条分支。玩家选择回应时，不是"选对选项"，而是"这个回应反映了我当前的状态"。
2. **雨夜压力系统：** 雨量 = f(游戏内时间, 玩家信念值)。信念越低，雨越大。大雨增加"倦怠"值（每 30s +1），迫使玩家找地方避雨。避雨触发回忆/思考场景。
3. **3 个月时钟：** 游戏内 3 个月 = 现实 90 分钟左右。每次重要交互消耗游戏内 1-3 天。每周出现一个"deadline 提醒"事件。第 3 个月结束时触发终局路线。

**Pros:**
- 与主题高度绑定 — 希望/绝望就是游戏的主题，不是附加的数值
- 极乐迪斯科已验证"系统即叙事"可行且受市场认可
- 状态即世界观滤镜而非 HP/MP，避免了"受伤回血"的 RPG 俗套
- 三组滑条的互动产生丰富叙事可能性

**Cons:**
- 设计复杂，所有场景文本需准备至少 3 种状态变体
- 玩家可能感觉"被状态推着走"，缺乏主动选择感
- 对话检定如果只依赖状态值，玩家可能觉得"我的选择不重要"
- 3 个月时钟可能造成不必要的"赶时间"焦虑

**Risk:** Medium — 设计理念成熟但实现复杂度高，需大量文本迭代

**Effort:** Large — 3-4 周设计 + 2 周原型验证

---

### Approach B: 简化版 — 单轴选择因果系统

**Description:**

放弃三轴系统，改用单一「希望值」+ 事件因果图。核心逻辑：每个场景让玩家做 1-2 个关键选择，每个选择直接增加或减少希望值。希望值影响的是结局（而非过程）。环境不随状态变化。

**Pros:**
- 实现简单，文本工作量小（每个场景只需 2 套结局文本）
- 玩家选择感强 — 每一步都知道"这个选择会改变结局"
- 测试和调试成本低
- 适合 3 个月开发周期的限制

**Cons:**
- 缺乏"世界在反映你内心"的沉浸感
- 单轴太简单，无法模拟"一个人的复杂性"
- 雨夜只是背景不是系统 — 失去了"环境即叙事"的设计优势
- 容易变成"选 A 加血选 B 扣血"的传统 RPG

**Risk:** Low — 实现简单但设计深度不足

**Effort:** Small-Medium — 1-2 周

---

### Approach C: 叙事权重系统 — 无滑条纯 flag

**Description:**

不用数值滑条，全部基于二元 flag 和触发器。每个 NPC 对话积累"理解碎片"和"情感碎片"。碎片达到阈值触发特定叙事事件。没有持续性状态，只有"当前剧情阶段"。

**Pros:**
- 最易实现，纯 flag 判断
- 玩家每次选择都产生明确的"剧情变化"
- 无数值管理负担

**Cons:**
- 无法模拟"逐渐绝望"的过程 — flag 是离散的，滑条是连续的
- "雨夜越来越大"这种渐变效果无法实现
- 没有持续压力感
- 与极乐迪斯科的设计哲学背道而驰

**Risk:** Low-Medium — 但设计方向与项目愿景冲突

**Effort:** Small — 1 周

---

### Recommendation

→ **Approach A（极乐迪斯科式三轴系统）** 因为：

1. 项目核心命题是"一个人的希望值在雨夜中如何被不同的人和经历改变" — 这天然需要连续性状态
2. 极乐迪斯科的方法论已被验证为 CRPG 领域的最佳实践
3. 三轴系统不是"加了更好"，而是"不加就无法实现" — 去掉滑条，主题就变成了传统的"讲故事"，而非"你的内心状态在讲故事"
4. Issue #4（状态-世界反馈）和 Issue #2（叙事架构）都直接依赖这套机制
5. 3 个月开发周期反而不矛盾 — 三轴系统**让有限内容产生最大变化感**：同一段对话，在不同状态值下呈现不同文本，相当于内容量翻倍

**但需接受 Approach A 的高复杂度，并在 PRD 中指出风险缓释措施：**
- 原型阶段先实现简化版（只做 2 组滑条，第 3 组后续添加）
- 对话检定先做文本模板替换，不做全量变体
- 3 个月时钟做成可开关选项，先默认关闭（固定 14 天，每条对话消耗 0.5-1 天）

## 5. Boundary Conditions & Acceptance Criteria

### 5.1 三组状态与叙事主题对应

| 滑条对 | 叙事对应 | 影响维度 |
|--------|----------|----------|
| 希望 ↔ 绝望 | 玩家对"3 个月做卖座游戏"的成败预期 | 世界呈现滤镜（暖/冷文字） |
| 热情 ↔ 倦怠 | 玩家对游戏开发的行动力和创造力 | 交互可能性和内容获取 |
| 信念 ↔ 动摇 | 玩家对自己"是否属于这个行业"的自我认同 | 检定权重和内心独白风格 |

**状态流转规则：**
- 初始值：希望 5 / 热情 5 / 信念 5（和 = 10，但只有一条轴公开）
- 状态不能对角翻转（希望不能直接变绝望，必须经过中间值 5）
- 每条轴朝一个方向移动后，需要"触发事件"才能折返

### 5.2 关键选择类型

- **道德选择（Moral）：** 选择如何对待他人 — 帮一个陌生人 vs 赶路。影响「希望」轴。例如："给流浪汉买咖啡"→ 希望 +1，"无视他继续走"→ 希望 -0.5（冷漠愧疚）。
- **生存选择（Survival）：** 选择如何对待自己 — 继续加班 vs 回家睡觉。影响「热情」轴。例如："再改一轮 bug"→ 热情 -1（倦怠），"先回家休息"→ 热情 +0.5（恢复但不进则退）。
- **信念选择（Conviction）：** 选择如何看待自己的身份 — "我是游戏开发者"vs"我可能不适合这行"。影响「信念」轴。例如：NPC 问"你为什么还在做这个？"，回答"因为我想做"→ 信念 +1，回答"因为没有别的选择"→ 信念 -1。

### 5.3 3 个月 deadline 的机制体现

**时钟机制（可开关）：**
- 1 个游戏月 ≈ 30 分钟现实时间
- 游戏从"第 1 个月"开始，在地铁站相遇时是"第 3 个月末"
- 每个场景/对话消耗 0.5-3 天（取决于内容长度）
- 每周触发一次 deadline 提醒（内心独白或环境变化）
- 第 3 个月到达时强制进入终局路线

**若不启用时钟：**
- 游戏仍然感知"压力"但是非线性的
- 雨夜强度作为替代压力系统
- 玩家可以自由探索所有场景

### 5.4 完整示例场景拆解

**场景："深夜便利店"**

1. **主题 → 机制：** 一个人深夜加班后在便利店买咖啡，遇到同样失业的同行。主题：同是天涯沦落人的希望感。
2. **状态值预设：** 希望=3（接近绝望），热情=6（还在全力干活），信念=4（开始动摇）
3. **环境呈现：** 希望=3 → "便利店的灯刺眼又苍白，收银机嗡嗡响像在嘲笑你"（冷色滤镜）
4. **对话检定：** 同行认出你（"你不是那个...做了三年的独立游戏？"）。触发「信念检定」：信念 ≥ 5 时，你可以坦然回答"还在做"；信念 < 5 时，你只能说"放弃了"。
5. **玩家选择：**
   - 选择 A（生存/信念）："还在做，这次能成" → 信念 +1，热情 +0.5
   - 选择 B（生存）："先找个外包活糊口" → 热情 +1（现实），信念 -0.5（妥协）
   - 选择 C（道德）：请对方喝咖啡 → 希望 +1，热情 +0.5
6. **世界反馈：** 希望从 3→4，便利店灯光描述变为"灯光微微泛黄"（暖了半度）
7. **后续影响：** 信念恢复到 4.5 → 下一个场景的内心独白从"我是不是该放弃"变为"也许再坚持一下"

### 5.5 映射链声明（每个提案机制）

**机制 A：对话即状态检定（Mapped Check）**
- 机制 → 玩家行为 → 主题表达
- 对话中隐性匹配当前状态值 → 玩家通过"怎么选"而非"选什么"表达内心 → 主题：你的**状态**定义你是谁，而非你的**知识**

**机制 B：希望绝望世界观滤镜**
- 机制 → 玩家行为 → 主题表达
- 状态值改变场景文字色调 → 玩家通过文字冷暖感知自己的内心变化 → 主题：**世界怎么呈现，取决于你怎么看**

**机制 C：雨夜压力系统**
- 机制 → 玩家行为 → 主题表达
- 信念值低 → 雨更大 → 被迫避雨 → 触发被迫停下的内心对话 → 主题：**环境是你内心的外化，无处可逃**

**机制 D：3 个月 deadline 时钟**
- 机制 → 玩家行为 → 主题表达
- 每场对话消耗游戏天数 → 玩家感受到时间流逝的紧迫 → 主题：**做游戏是在和时间赛跑，而终点不一定是成功**

**机制 E：三轴滑条结构**
- 机制 → 玩家行为 → 主题表达
- 三个维度变化而非单一数值 → 一个选择可能提升希望但降低信念 → 主题：**人的状态是复杂的，拯救了一面的同时可能失去另一面**

### 5.6 反向验证（去掉机制主题是否还在）

| 机制 | 去掉后 | 主题还在吗？ |
|------|--------|-------------|
| 对话即检定 | 对话变成纯叙事树，和传统 RPG 无异 | ✗ 主题消失 — 失去了"你的状态影响你怎么看这个世界"的核心体验 |
| 世界观滤镜 | 世界渲染是静态的，玩家内心变化无外在反馈 | ✗ 主题消失 — 失去了"内心外化"的设计核心 |
| 雨夜压力 | 雨只是背景，没有"雨夜会改变你"的感觉 | △ 部分保留 — deadline 可替代压力感 |
| 3 个月时钟 | 去掉后没有紧迫感，但叙事依然成立 | △ 部分保留 — 故事依然关于一个困境中的人 |
| 三轴滑条 | 变成单值"希望值"，选择变成简单的加减法 | △ 部分保留 — 但失去了"复杂人格"的深度 |

**结论：** 最核心不可替代的是**机制 A（对话即检定）** 和**机制 B（世界观滤镜）** — 这两者去掉后，主题几乎完全消失。

### 5.7 替换测试（放到不同主题游戏）

| 机制 | 放到不同主题游戏 | 合理吗？ |
|------|-----------------|---------|
| 对话即检定 | 放到《最终幻想》类 JRPG | ✗ 不合理 — JRPG 需要明确的选择分支和奖励反馈 |
| 对话即检定 | 放到《极乐迪斯科》 | ✓ 完全合理 — 这正是该游戏的机制 |
| 世界观滤镜 | 放到《寂静岭》 | △ 合理但目标不同 — 心理环境映射概念相通 |
| 雨夜压力 | 放到《赛博朋克2077》 | ✓ 合理但太浅 — 环境压力是通用机制 |
| 3 个月时钟 | 放到《女神异闻录》 | ✓ 完全合理 — 时间管理是该游戏核心 |
| 三轴滑条 | 放到《巫师3》 | ✗ 不合理 — 该游戏需要明确的数值升级 |

**结论：** 这组机制整体与"CRPG + 心理叙事"主题高度绑定，不适合套用到传统 RPG。

| **总分** | 21 | 19 | 19 | 22 | 20 |

### 5.9 被淘汰的候选机制及淘汰理由

| 候选机制 | 淘汰理由 |
|---------|---------|
| **单轴纯 flag 系统**（Approach C） | 无法模拟"逐渐绝望"的渐变过程，与项目核心命题冲突 |
| **传统 RPG 属性点**（力量/智力/敏捷） | 与主题完全无关 — 这是关于内心的故事，不是战斗 |
| **技能树/天赋盘** | 不符合"3个月绝望期"设置 — 玩家不是变强，是在变或不变 |
| **经营模拟元素**（管理现金流/团队） | 偏离"内心叙事"的文学性定位 |
| **QTE 事件** | 破坏海明威式节奏和文学沉浸感 |
| **多分支对话树**（传统 CRPG 方式） | 需要大量文本，3 个月开发周期不可行 |
| **声望系统**（NPC 好感度） | 核心不是他人怎么看你，是你怎么看自己 |

### 5.10 结合度最弱的机制及改进方案

**最弱机制：世界观滤镜（玩法深度仅 2 分）**

**问题：** 世界观滤镜本质上是文本替换 + 文字描述变换。它极大地丰富了叙事体验，但"可玩性"很低 — 玩家只是在被动接收不同的文字，没有主动"操作"的空间。

**改进方案：**
1. **加入主动感知机制：** 玩家可以主动"凝视"场景中的特定对象（长按交互键），在当前滤镜下获得独特的内心独白。例如：高绝望时凝视便利店灯光 → "光太冷了，像是医院走廊"；高希望时同一盏灯 → "光暖洋洋的，让人想多待一会儿"。
2. **加入"滤镜切换"微交互：** 玩家可以在对话中短暂"切换滤镜"来理解 NPC 的另一面。比如：用高信念视角看 NPC → 看到对方的坚定；用高动摇视角看 → 看到对方的不安。这既是玩法也是叙事。
3. **增加感知反馈变化曲线：** 状态变化 1-2 点时文字不变，3 点时才变化。让玩家主动感知"我好像和刚才不一样了"。

**评分改进：** 玩法深度 2 → 3.5，总分 19 → 20.5

## 7. Spike / Experiment

> **depth/deep 必备章节**

### Question to Answer

**核心不确定性：三组滑条能否在"3 个月内"（有限的文本量下）产生足够的叙事变化感？**

具体来说：
1. 每个场景需要多少变体文本才能"感觉不一样"？
2. 对话检定用区间匹配还是精确值匹配更有效？
3. 3 个月时钟的"消耗速度"应该多快才能产生紧张感又不破坏探索？

### Method

**Spike 1: 文本变体密度实验**
- 写一个迷你场景（便利店）的 6 个文本变体：高希望/中/低 × 高信念/低
- 估算每个场景的平均变体字数
- 计算 7 个场景总变体量

**Spike 2: 检定区间原型**
- 在 Godot 中写一个最小 prototype：`state_system.gd`（3 个 int 变量 + 信号）+ 一个 Label 显示当前"滤镜描述"
- 模拟对话选择 → 状态变化 → 文字切换
- 验证 GDScript 2.0 的 `@export` 和信号广播机制是否满足需求

**Spike 3: 时钟消耗模拟**
- 枚举 7 个场景 + 子对话的数量
- 计算不同的"消耗速度"下，玩家多少分钟到达终局
- 目标是 60-90 分钟通关，约 4-6 个关键对话

### Result

**Spike 1 预期结果：**
- 每个场景约需要 3-5 段变体文本（场景描述 + NPC 语气 + 对话回响）
- 每段变体约 20-40 字（海明威约束）
- 7 个场景总变体 ≈ 600-1400 字
- 加上基准文本，总文本量 ≈ 3000-5000 字 — **在 3 个月开发周期内可行**

**Spike 2 预期结果：**
- `state_changed` 信号 + 场景文本切换基本模式可验证
- 阈值区间判定（而非精确值匹配）更适合 CRPG — 减少变体需求

**Spike 3 预期结果：**
- 7 场景 × 2-3 关键对话 = 14-21 个主要交互点
- 每个交互消耗 2-4 天 → 28-84 天 → 约 1-3 个月游戏内
- 建议消耗速度：常规对话 1 天，关键抉择 3 天
- 时钟开启情况下，玩家约 40-60 分钟到达终局 — 适当

### Impact on Approach

| 发现 | 影响 |
|------|------|
| 文本变体量在 3 个月开发周期内可行 | → 验证 Approach A 可行性 |
| 阈值区间优于精确值 | → 对话条件系统用区间判定 |
| 时钟消耗 1 天/常规对话 | → 时钟默认开启，消耗速度可调 |
| 整体文本量 ~5000 字 | → 海明威约束下，文本创作是主要工作量 |
